# EduPulse 🎓

**EduPulse** — персональный рабочий инструмент преподавателя детских кружков.

> Не CRM. Не журнал. **Чек-лист преподавателя.**

Приложение отвечает не на вопрос «Какие у меня есть сущности?», а на вопрос **«Что мне нужно сделать сегодня?»**

Единственный пользователь EduPulse — преподаватель (или администратор центра).
Никаких кабинетов родителей, никаких уведомлений им, никаких лишних сущностей.

## Философия

### От проектирования программы — к проектированию рабочего дня

Большинство CRM для школ построены вокруг **сущностей**: курсы, ученики, платежи. Пользователь сам собирает пазл:

> «Так, у меня сегодня Junior в 17:00… надо проверить, кто пробный… а у Маши вроде оплата…»

EduPulse построен иначе.

```
У преподавателя есть день.

В течение дня появляются задачи.

EduPulse помогает закрыть эти задачи.
```

Приложение не требует от преподавателя «управлять базой данных». Оно показывает, что нужно сделать, и помогает сделать это быстро.

### Две части приложения

**Работа** — ежедневный цикл, открывается 20 раз в день:
```
Inbox → Lesson → Student
```

**Управление** — административная часть, открывается редко:
```
Школа → Курсы → Ученики → Платежи
```

### Lesson — сердце EduPulse

Inbox только приводит преподавателя к работе.
**Настоящее сердце EduPulse — это Lesson.**

Проведение занятия включает несколько действий:
1. Открыть занятие
2. Отметить присутствующих
3. Добавить пробного ученика, если пришёл
4. Оставить заметку
5. При необходимости записать оплату
6. Закрыть занятие

Страница **Lesson** — это не просто «экран посещаемости». Это **рабочее место преподавателя на время конкретного урока**.

## Сценарий дня преподавателя

### Утро. Открыть Inbox

Преподаватель открывает EduPulse → видит **Inbox** — задачи, сгруппированные по категориям.

```
📥 Сегодня, среда, 8 июля

📚 Занятия (2)
  ↓ 17:00 Junior · 2 не отмечены
  ↓ 19:00 Scratch · Все отмечены

💰 Оплаты (3)
  ↓ Анна · Просрочена
  ↓ Миша · Нет оплат

🌟 Пробные (1)
  ↓ Артём · Новый ученик
```

Каждый пункт завершается действием. Не информацией — действием.

### Занятие. Lesson — рабочее место

Нажал на занятие → **LessonDetailPage**:

```
🕐 17:00 · Робототехника Junior

Ученики:
✅ Был     — Иван Петров
🌟 Пробное — Артём Сидоров  (добавлен прямо на занятии)
⏰ Опоздал — Екатерина Волкова
❌ Не был  — Максим Кузнецов · болеет

📝 Домашнее задание
📍 Адрес и ссылка в навигаторе
```

Каждый статус — один тап:
- ✅ **Был** — зелёный, основной
- 🌟 **Пробное** — фиолетовый
- ⏰ **Опоздал** — янтарный
- ❌ **Не был** — красный, с комментарием

Можно добавить нового ученика прямо на занятии → он автоматически пробный.

### В течение дня. Быстрые действия

- Добавить нового ученика
- Записать оплату (прямо из Inbox или карточки ученика)
- Посмотреть контакты родителя

### Конец дня или редко. Статистика

**📊 Статистика** — сколько занятий проведено, посещаемость, оплаты. Открывается не каждый день, но даёт общую картину.

### Редко. Карточка ученика

Карточка — **рабочая карточка преподавателя**. Открывая её, преподаватель быстро отвечает себе:

- Кто это?
- На какой курс ходит?
- Как связаться с родителем?
- Когда последний раз был?
- Есть ли задолженность?
- Какие заметки?

### Ключевые архитектурные решения

1. **Lesson-centric** (а не Course-centric) — занятие — главная сущность. Курс — шаблон расписания.
2. **Action-first** — главный экран не дашборд, а Inbox с действиями.
3. **Группировка** — Inbox группирует сигналы по категориям (занятия, оплаты, пробные), а не смешивает в плоский список.
4. **Google Sheets как база** — прозрачные данные, редактируемые вручную при необходимости.
5. **In-memory fallback** — для демо и разработки без Google Sheets.
6. **Единый API-клиент** — весь фронтенд общается с бэком через `api.ts`. Переход на PostgreSQL затронет только backend.

## Модель владения данными

### Принципы

1. **Каждая запись принадлежит одному пользователю.** Поле `user_id` (внутренний ID, например `usr_A81K2P`) хранится во всех таблицах: `Students`, `Courses`, `Lessons`, `Attendance`, `Payments`, `Achievements`.
2. **Backend сам проставляет `user_id`.** Фронтенд никогда не отправляет `user_id` — сервер берёт его из JWT-токена текущего пользователя.
3. **Backend сам фильтрует запросы.** Сервисы получают `current_user` и решают, какие данные возвращать:
   - **Owner** — все данные всех пользователей (через те же сервисы, не в обход)
   - **User** — только свои записи (`user_id == current_user.user_id`)
   - **Tester** — без автоматического доступа к реальным данным пользователей
4. **Owner — владелец платформы, а не владелец данных.** Owner управляет системой (роли, статистика, диагностика), но не учениками других преподавателей.

### Схема данных

```
Users
├── id: usr_A81K2P           ← внутренний, неизменяемый идентификатор
├── telegram_id: 123456789    ← способ входа (может быть + email, Google, Apple)
├── role: owner | admin | user | tester
├── is_active: bool
├── created_at
└── updated_at

UserSettings                   ← будущие настройки (timezone, language, theme...)
└── user_id

Students     → user_id        ← все сущности хранят user_id владельца
Courses      → user_id
Lessons      → user_id
Attendance   → user_id
Payments     → user_id
Achievements → user_id
```

### Поток аутентификации

```
Telegram Login
      │
      ▼
telegram_id          ← внешний идентификатор (может измениться)
      │
      ▼
Users.telegram_id    ← находим пользователя
      │
      ▼
user_id (usr_...)    ← внутренний ID (никогда не меняется)
      │
      ▼
Вся система работает через user_id
```

### Роли

| Роль | Описание |
|------|----------|
| **Owner** | Владелец платформы. Полный доступ + системная панель (`/owner`). Проверяется по `OWNER_TELEGRAM_ID` из `.env` |
| **Admin** | Зарезервировано для будущего (администраторы учебных центров). В текущей версии не используется |
| **User** | Преподаватель. Создаёт и управляет **только своими** учениками, курсами, занятиями, оплатами, достижениями |
| **Tester** | Режим тестирования. Без автоматического доступа к реальным данным пользователей |

### Почему не telegram_id как ключ

- `telegram_id` — внешний идентификатор. Завтра может появиться Google Login, Email, Apple ID.
- `user_id` — внутренний, никогда не меняется. Все foreign keys ссылаются на него.
- Если менять провайдера аутентификации, достаточно обновить одну строку в `Users`, а не все таблицы.

### Почему не owner_id

Слово **Owner** уже занято ролью владельца платформы. `user_id` однозначно обозначает «владелец записи» без путаницы с ролью.

### Будущее: учебные центры

Архитектура готова к добавлению `workspace_id` / `organization_id` без переписывания:

```
Organization
      │
      ├── User
      │      ├── Students (user_id + workspace_id)
      │      └── Courses  (user_id + workspace_id)
      │
      └── User
             ├── Students
             └── Courses
```

Сейчас `workspace_id` не добавляется — только `user_id`.

## Быстрый старт

```bash
# Требования: Node.js 20+, Python 3.12+

# 1. Frontend
cd frontend
npm install
npm run dev                 # http://localhost:5173

# 2. Backend (в другом терминале)
cd backend
python -m venv venv
# source venv/bin/activate  # Linux/Mac
# venv\Scripts\activate     # Windows
pip install -r requirements.txt
cp .env.example .env        # Отредактируйте .env
uvicorn app.main:app --reload  # http://localhost:8000

# 3. Или всё сразу через Docker
docker compose up --build   # http://localhost:80
```

## Структура данных

### Связи между сущностями

```
Course (шаблон расписания)
├── title, days[], time, price, color
└── → Groups[]                  ← группы курса

Group (устойчивая группа учеников)
├── name, days[], start_time, end_time
├── student_ids[]               ← постоянный состав группы
├── location, teacher
└── → Lessons[]                 ← авто-создаются из расписания группы

Lesson (конкретное занятие) ★
├── course_id, group_id, date, time, title
├── status: scheduled | completed | cancelled | rescheduled
├── homework, location, note
├── Attendance[]                ← отметки по lesson_id
└── Ученики наследуются из группы (group.student_ids)

Student
├── first_name, last_name, возраст
├── contact                     ← контактное лицо (без кабинета родителя)
├── course_ids[]               ← на каких курсах состоит
└── Attendance[]               ← история посещений

Payment (журнал — без расчётов)
├── student_id, course_id, amount, payment_date
├── payment_type               ← тип для удобства (monthly, partial и т.д.)
└── comment                    ← «Оплата за июль» — пишет человек
```

**Ключевые отличия от типичной CRM:**
- Родитель — это **контактное поле** в карточке ученика, а не отдельный пользователь. Никаких кабинетов, логинов, уведомлений для родителей.
- Нет `payment_summary`, `balance`, `required_amount`, `paid_until`, `progress` — только история.
- Attendance привязана к `lesson_id` (конкретному занятию), а не к курсу + дате.

### Таблицы Google Sheets (7 листов)

**Users** — `id`, `telegram_id`, `first_name`, `last_name`, `username`, `photo_url`, `role`, `is_active`, `created_at`

**Courses** — `id`, `title`, `description`, `days`, `time`, `price`, `teacher_id`, `color`, `student_ids`, `location`, `location_link`, `is_active`, `created_at`, `monthly_price`, `lesson_price`, `lessons_per_week`, `payment_type`

**Students** — `id`, `first_name`, `last_name`, `age`, `birth_date`, `parent_contact`, `parent_name`, `parent_relation`, `phone`, `telegram`, `course_ids`, `start_date`, `photo_url`, `is_active`, `created_at`

**Lessons** — `id`, `course_id`, `group_id`, `date`, `time`, `start_time`, `end_time`, `title`, `status`, `rescheduled_to`, `homework`, `location`, `location_link`, `note`, `lesson_type`, `is_active`, `created_at`

**Attendance** — `id`, `lesson_id`, `date`, `course_id`, `student_id`, `status`, `comment`, `marked_by`, `created_at`

**Payments** — `id`, `student_id`, `course_id`, `amount`, `payment_date`, `payment_type`, `comment`, `created_at`

**Achievements** — `id`, `student_id`, `title`, `icon`, `description`, `achieved_at`, `created_at`

## Inbox — главный экран

Inbox — это не дашборд и не список уведомлений. Это **чек-лист преподавателя на сегодня**, сгруппированный по категориям.

### Группы сигналов

| Группа      | Иконка | Откуда берётся                            | Пример                             |
|-------------|--------|-------------------------------------------|------------------------------------|
| 📚 Занятия  | 📚     | Занятия на сегодня с неотмеченными детьми | «17:00 Junior · 2 не отмечены»     |
| 💰 Оплаты   | 💰     | Просроченные платежи (>30 дней)           | «Анна · Просрочена»                |
| 🌟 Пробные  | 🌟     | Новые ученики без посещений               | «Артём · Новый ученик»             |
| ⚠ Требуют  | ⚠      | Длительное отсутствие (>14 дней)          | «Максим · Не был 18 дней»          |
| внимания    |        |                                           |                                    |


### Почему группировка, а не список приоритетов

Со временем у преподавателя может быть 8 курсов, 140 учеников, 40 оплат. Плоский список приоритетов превратится в свалку. Группировка по категориям позволяет:

- Быстро оценить объём работы по каждой категории
- Скрыть неактуальные группы
- Не терять важное в шуме

### Как работает

1. При загрузке `GET /api/inbox` собирает все сигналы
2. Группирует по категориям: занятия, оплаты, пробные, требующие внимания
3. Внутри группы сортирует по приоритету
4. Каждый элемент — кликабельная карточка с действием
5. Данные обновляются каждые 30 секунд
6. Группы можно сворачивать

### Почему это не дашборд

Дашборд показывает **цифры**:
```
Занятий: 3 · Учеников: 12 · Пришли: 8/12 · Оплаты: 2
```

Inbox показывает **действия**:
```
📚 Занятия (2)
  ↓ 17:00 Junior → 2 не отмечены → нажмите, чтобы отметить
  ↓ 19:00 Scratch → Все отмечены

💰 Оплаты (3)
  ↓ Анна → Просрочена → Добавить платеж
```

## Lesson — сердце приложения

Inbox приводит преподавателя к работе. **Lesson — место, где работа делается.**

### Что можно сделать на странице занятия

1. **Отметить посещаемость** — один тап по статусу (✅ был, 🌟 пробное, ⏰ опоздал, ❌ не был)
2. **Добавить ученика** — нового (сразу как пробное) или существующего с другого курса
3. **Добавить комментарий** к отметке (например, «болеет»)
4. **Домашнее задание** — добавить и сохранить
5. **Сменить статус занятия** — провести, отменить, перенести
6. **Локация** — адрес и ссылка в навигатор

Lesson объединяет все действия, которые преподаватель выполняет во время и вокруг конкретного занятия.

## Lesson-centric архитектура

### Проблема Course-centric

Раньше всё было завязано на курсе:

```
Course → Attendance (course_id + date)
```

Нельзя было:
- Отменить конкретное занятие
- Перенести его на другой день
- Прикрепить домашнее задание
- Добавить ученика «только на сегодня» (пробное) без записи на весь курс

### Решение

```
Course (шаблон расписания)
├── title, days[], time, price
├── student_ids[] ← постоянные ученики
└── → Lessons[] ← авто-создаются

Lesson (экземпляр занятия) ★
├── id, date, time, title
├── status: scheduled | completed | cancelled | rescheduled
├── homework, location, note
└── Attendance[] ← по lesson_id
```

### Автогенерация

Когда преподаватель открывает Inbox или страницу занятия:
1. Проверяется: есть ли Lesson для Course + сегодняшняя дата?
2. Если нет → создаётся Lesson (status=scheduled, time=course.time)
3. Если есть → открывается существующий

Вручную можно создать разовое занятие (отработка, открытый урок).

## Аутентификация

### Telegram Mini App

- Вход через `telegram-web-app.js` initData
- HMAC-SHA256 верификация через bot token
- Защита от replay-атак (проверка `auth_date`)

### JWT

- HS256, 24 часа, роль в токене
- Bearer-схема в `Authorization`
- `CurrentUser` — FastAPI dependency для всех защищённых эндпоинтов

### Режим демо

`DEBUG=true` без Google Sheets:
- Демо-данные в памяти (3 курса, 4 ученика, отметки, платежи, достижения)
- Авторизация отключена (возвращается Admin)
- Данные сбрасываются при перезапуске
